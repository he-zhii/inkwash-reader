# 🧠 DeepRead — AI 深度阅读助手 · 产品设计方案

> **版本**: v1.0 Draft
> **日期**: 2026-05-11
> **定位**: 个人使用的 AI 辅助阅读工具，集成于现有「碧树西风文集」系统

---

## 一、产品定位与核心理念

### 你遇到的真实问题

| 痛点 | 本质 |
|------|------|
| 文章太长，读到后面忘了前面 | **认知负荷过高**，缺乏结构化导航 |
| 缺乏背景知识，某些段落看不懂 | **知识断层**，需要即时补充上下文 |
| 读完了但感觉没读透 | 缺乏**主动思考引导**，被动接收信息 |
| 想讨论但没有对话对象 | 缺乏**即时的智识对话伙伴** |

### 产品愿景

> **DeepRead 不是一个「帮你读」的工具，而是一个「陪你读」的伙伴。**

核心设计哲学：
- 🎯 **不替代阅读，而是降低阅读门槛** — 你仍然是主角
- 🧭 **先全局再局部** — 先建立框架感，再深入细节
- 💬 **从被动接收到主动探索** — 鼓励提问和讨论
- 🪶 **极简交互** — 一个人用的工具，不需要复杂 UI

## 二、核心功能模块设计

### 模块一：文章 X 光 (Article X-Ray)

> 一键透视文章骨架，建立全局认知地图

```
┌─────────────────────────────────────────────────────────┐
│  📖 文章正文                    │  🧠 X-Ray 面板        │
│                                 │                       │
│  [段落 1 正文内容...]           │  📋 文章摘要          │
│  ─────────────────              │  "本文讨论了..."       │
│  [段落 2 正文内容...]           │                       │
│  ─────────────────              │  📊 段落导航          │
│  [段落 3 正文内容...]           │  ┌─────────────────┐  │
│  ─────────────────              │  │ ① 开篇引入 ← 你在这│
│  [段落 4 正文内容...]           │  │ ② 核心论点      │  │
│                                 │  │ ③ 案例分析      │  │
│                                 │  │ ④ 深层推演      │  │
│                                 │  │ ⑤ 总结收束      │  │
│                                 │  └─────────────────┘  │
│                                 │                       │
│                                 │  🏷️ 关键概念          │
│                                 │  #认知偏差 #决策模型   │
│                                 │                       │
└─────────────────────────────────────────────────────────┘
```

#### 功能拆解

| 子功能 | 描述 | 交互方式 |
|--------|------|----------|
| **全文摘要** | 3-5 句话概括文章核心论点和结论 | 自动生成，展示在面板顶部 |
| **段落导航** | 每段提炼一句话梗概，形成可点击的目录 | 点击跳转到对应段落，当前段落高亮 |
| **阅读进度感知** | 实时追踪你读到了哪一段 | 滚动自动更新，导航面板同步高亮 |
| **关键概念提取** | 提取文中出现的核心概念/术语 | 标签形式展示，点击可快速问 AI |
| **难度预估** | 预估文章的阅读难度和所需时间 | 用 ⭐ 表示难度，标注预估阅读时间 |

#### 段落导航的定义逻辑

不是严格按照 HTML 的 `<p>` 标签来划分，而是按**语义段落**：
- 一个论点 + 其展开 = 一个语义段落
- 连续的例证/数据 = 归并为一个段落
- 转折/递进 = 新段落的开始

这样做的好处：避免段落过碎，保持导航的可读性（目标控制在 5-12 个段落）。

### 模块二：深度对话 (Deep Talk)

> 选中任意文本，与 AI 展开即时的深度讨论

#### 2.1 触发方式

```
用户选中一段文字
        ↓
 ┌──────────────────────┐
 │ 🔍 解释  📝 举例     │  ← 浮动工具栏 (Floating Toolbar)
 │ 🧩 拆解  💬 讨论     │
 └──────────────────────┘
```

四种快捷操作：

| 操作 | 说明 | 适用场景 |
|------|------|----------|
| 🔍 **解释** | 用通俗语言解释选中内容的含义 | 看不懂某句话/术语 |
| 📝 **举例** | 用具体的生活案例来说明这个概念 | 理解了字面意思但缺乏直觉 |
| 🧩 **拆解** | 深度拆解选中内容的逻辑结构和推理链 | 想搞清楚作者是怎么论证的 |
| 💬 **讨论** | 开启一个自由对话，基于选中内容聊 | 有自己的想法，想碰撞一下 |

#### 2.2 对话面板设计

```
┌─────────────────────────────────────────┐
│  💬 Deep Talk                     ✕     │
│─────────────────────────────────────────│
│  📌 选中内容:                           │
│  "人们往往高估短期变化的影响，             │
│   低估长期变化的影响。"                   │
│─────────────────────────────────────────│
│                                         │
│  🤖 AI:                                │
│  这句话被称为"阿马拉定律"(Amara's Law)  │
│  ...                                    │
│                                         │
│  👤 你:                                │
│  能举一个科技领域的例子吗？              │
│                                         │
│  🤖 AI:                                │
│  当然。以智能手机为例...                  │
│                                         │
│─────────────────────────────────────────│
│  [输入你的想法...]              发送 ▶   │
│  💡 追问建议: "作者为什么在这里提到这个?" │
└─────────────────────────────────────────┘
```

#### 2.3 AI 对话的关键设计原则

1. **全文上下文注入** — AI 始终持有完整文章作为背景知识，不是只看选中片段
2. **位置感知** — AI 知道选中内容在文章的哪个位置，能联系前后文
3. **追问引导** — 每次回答后，AI 主动建议 1-2 个追问方向，降低提问门槛
4. **对话记忆** — 同一篇文章内的对话历史保持连续，不会每次重新开始
5. **风格适配** — 回答风格偏向"聪明的朋友给你讲解"，不是教科书式

## 三、交互设计细节

### 3.1 整体布局

```
┌──────────────────────────────────────────────────────────────┐
│  Header: 碧树西风文集  ·  文章标题          [X-Ray] [主题]   │
├──────────────────────────────────────────────────────────────┤
│                        │                                     │
│   📖 文章正文           │   📊 X-Ray 面板                     │
│   (主阅读区)           │    ↕ 可折叠                         │
│                        │   或                                │
│   选中文字 → 浮动栏    │   💬 Deep Talk 面板                  │
│                        │    (两个面板可切换)                   │
│                        │                                     │
├──────────────────────────────────────────────────────────────┤
│  Reading Progress Bar                                        │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 面板切换逻辑

| 场景 | 行为 |
|------|------|
| 打开文章页 | 默认显示 X-Ray 面板（如果已生成），否则显示「生成分析」按钮 |
| 选中文本并点击操作 | 自动切换到 Deep Talk 面板，X-Ray 面板暂时隐藏 |
| 关闭 Deep Talk | 回到 X-Ray 面板 |
| 点击 X-Ray 中的概念标签 | 自动打开 Deep Talk 并以该概念发起提问 |
| 窗口较窄时 | 右侧面板变为底部抽屉式弹出 |

### 3.3 首次分析流程

```
用户打开文章 → 右侧面板显示：
 ┌────────────────────────┐
 │                        │
 │    🧠 AI 阅读助手       │
 │                        │
 │  点击下方按钮，AI 将：    │
 │  · 生成全文摘要          │
 │  · 梳理段落结构          │
 │  · 提取关键概念          │
 │                        │
 │  ┌──────────────────┐  │
 │  │  ▶ 开始分析       │  │
 │  └──────────────────┘  │
 │                        │
 │  ⏱ 预计 5-10 秒        │
 └────────────────────────┘

点击后 → 流式输出分析结果
分析完成 → 缓存结果，下次打开直接展示
```

### 3.4 滚动联动设计

当用户滚动文章时：
1. X-Ray 面板中的段落导航**实时高亮**当前正在阅读的段落
2. 高亮段落自动滚动到面板可视区域内（类似 VS Code 的 minimap 行为）
3. 已读过的段落显示 ✓ 标记（可选）

## 四、技术架构方案

### 4.1 系统架构

```
┌─────────────────────────────────────────────────┐
│                  浏览器前端                       │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 文章渲染  │  │ X-Ray UI │  │ Deep Talk UI  │  │
│  │ (现有)    │  │ (新增)   │  │ (新增)        │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│        ↓              ↓              ↓           │
│  ┌──────────────────────────────────────────┐    │
│  │          AI Service Layer (JS)            │    │
│  │  · API 调用封装                           │    │
│  │  · 流式响应处理 (SSE)                     │    │
│  │  · 上下文管理 & Prompt 工程               │    │
│  │  · 本地缓存 (LocalStorage / IndexedDB)   │    │
│  └──────────────────────────────────────────┘    │
│                       ↓                          │
└───────────────────────│──────────────────────────┘
                        ↓
              ┌────────────────────┐
              │   LLM API 服务     │
              │  (方案 A / B / C)  │
              └────────────────────┘
```

### 4.2 LLM 接入方案对比

| 方案 | 描述 | 优势 | 劣势 | 推荐度 |
|------|------|------|------|--------|
| **A. 直连 API** | 前端直接调用 OpenAI / Claude / DeepSeek API | 最简单，无需后端 | API Key 暴露在前端（个人用可接受） | ⭐⭐⭐⭐⭐ |
| **B. 轻量代理** | 一个极简的 Node.js 代理层，转发请求并管理 Key | Key 安全，可加缓存 | 需部署一个小服务 | ⭐⭐⭐⭐ |
| **C. 本地模型** | 使用 Ollama + 本地模型 | 完全免费，隐私好 | 质量有限，需要好显卡 | ⭐⭐⭐ |

> **我的推荐：方案 A（直连 API）**
>
> 理由：只有你一个人用，API Key 写在前端的 JS 里或者存在 LocalStorage 里完全没问题。省去所有后端部署的复杂度，代码量最小，维护成本为零。
>
> 推荐模型：**DeepSeek V3**（中文能力强、成本极低、支持长上下文）或 **Claude 3.5 Sonnet**（综合能力最强）。

### 4.3 与现有系统的集成方式

你现有的 `pages_beautify` 项目结构：

```
pages_beautify/
├── scripts/build.js     ← 构建脚本（需要修改：注入 AI 模块代码）
├── src/
│   ├── app.js           ← 前端逻辑（需要扩展：加入 AI 功能）
│   └── style.css        ← 样式（需要扩展：加入面板样式）
├── dist/                ← 构建输出
└── *.md                 ← 文章源文件
```

集成策略 — **在现有系统上增量添加**，不重构：

| 新增文件 | 职责 |
|----------|------|
| `src/ai-service.js` | AI API 调用封装、Prompt 模板、流式处理 |
| `src/xray-panel.js` | X-Ray 面板的 UI 逻辑、段落导航、滚动联动 |
| `src/deeptalk-panel.js` | Deep Talk 面板的 UI 逻辑、对话管理 |
| `src/text-selection.js` | 文本选中检测 + 浮动工具栏 |
| `src/ai-style.css` | AI 功能相关样式 |

构建脚本修改：
- 在 `generateArticlePage()` 中注入 AI 面板的 HTML 骨架
- 合并新增 JS/CSS 到 `dist/`

### 4.4 数据缓存策略

| 数据类型 | 存储方式 | 过期策略 |
|----------|----------|----------|
| X-Ray 分析结果 | IndexedDB | 永不过期（文章内容固定） |
| Deep Talk 对话历史 | IndexedDB | 按文章保留，手动清除 |
| API Key | LocalStorage | 永不过期 |
| 用户阅读进度 | LocalStorage | 永不过期 |

## 五、Prompt 工程设计

### 5.1 X-Ray 分析 Prompt

```
你是一位专业的阅读分析助手。请分析以下文章，输出结构化的阅读辅助信息。

要求：
1. 全文摘要：用 3-5 句话概括文章的核心观点和结论
2. 段落分析：将文章按语义划分为 5-12 个段落，每个段落给出：
   - 起始标记（该段落第一句话的前 10 个字）
   - 一句话梗概（不超过 25 字）
   - 重要程度（核心/支撑/过渡）
3. 关键概念：提取 3-8 个核心概念或术语
4. 阅读难度：1-5 星
5. 预估阅读时间：X 分钟

请以 JSON 格式输出。

文章内容：
---
{article_content}
---
```

### 5.2 Deep Talk System Prompt

```
你是一位博学且善于表达的阅读伙伴。用户正在阅读一篇文章，
需要你帮助他深入理解文中的内容。

你的风格：
- 像一个聪明的朋友在跟他聊天，不是在上课
- 解释要通俗易懂，多用生活化的类比
- 举例要具体、生动、有画面感
- 如果用户的理解有偏差，温和地纠正
- 每次回答后，建议 1-2 个值得追问的方向

完整文章内容（作为你的知识背景）：
---
{article_content}
---

用户当前选中的文本位于文章的第 {position} 部分。
```

## 八、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 文章过长超出模型上下文窗口 | 分析不完整 | 使用 DeepSeek V3（128K 上下文），或按段落分批处理 |
| API 调用成本 | 长期使用的费用 | DeepSeek 极便宜（约 ¥0.01/篇文章分析）；缓存避免重复调用 |
| 分析质量不稳定 | 段落划分不合理 | 精调 Prompt + 结果校验；用户可手动触发重新分析 |
| 前端 API Key 安全 | 被他人使用 | 个人项目风险极低；可设置 API Key 用量限制 |

## 九、设计原则总结

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  1. 阅读第一，AI 第二                             │
│     → AI 面板不遮挡正文，随时可收起                │
│                                                 │
│  2. 按需出现，不打扰                              │
│     → 不自动分析，用户主动触发                     │
│     → 浮动栏只在选中文本时出现                     │
│                                                 │
│  3. 结果可信，来源可溯                            │
│     → 每个段落梗概可以跳到原文验证                  │
│     → 解释/举例始终基于原文上下文                   │
│                                                 │
│  4. 快速迭代，先跑起来                            │
│     → Phase 1 就能用，后续逐步完善                 │
│     → 不追求完美，追求实用                         │
│                                                 │
└─────────────────────────────────────────────────┘
```
